Архитектурные и технические решения
1. Шаблон архитектуры MVC
   
Cпособствует разделению задач: бизнес-логика, представление и доступ к данным отделены друг от друга.

Это повышает удобство обслуживания и тестирования: каждый компонент можно тестировать и модифицировать независимо.

Облегчает совместную работу в команде: разработчики могут работать на разных уровнях одновременно.

Это соответствует соглашениям Laravel и использует встроенные функции.

Реализация:

Модели (App\Models\*): представляют объекты данных и бизнес-правила (например, User, Request).

Контроллеры (App\Http\Controllers\*): обрабатывают HTTP-запросы, организуют бизнес-логику и возвращают ответы.

Представления (resources/views/): визуализация HTML-шаблонов с помощью Blade.

Хранилища (App\Repositories\*): абстрагирование логики доступа к данным от контроллеров.

Сервисы (App\Services\*): инкапсуляция сложных бизнес-процессов.

2. RequestRepository для доступа к данным.

Отделяет контроллеры от Eloquent ORM, упрощая переключение источников данных.

Централизует логику запросов в выделенных классах, улучшая организацию кода.

Упрощает модульное тестирование.

Обеспечивает повторное использование кода в нескольких контроллерах и службах.

Реализация: RequestRepository предоставляет методы, подобные getListBy(), для получения запросов с фильтрами.

3. Service layer для бизнес-логики (App\Services\*).

Позволяет снизить нагрузку на контроллеры и сосредоточить их на обработке запросов и ответов.

Обеспечивает возможность повторного использования бизнес-логики на нескольких конечных точках.

Улучшает тестируемость: сервисы могут быть протестированы в модульном режиме независимо от контекста HTTP.

Поддерживает управление транзакциями и сложные рабочие процессы (например, переход состояния с блокировкой).

Реализация: RequestService обрабатывает такие операции, как create(), assign() и UpdateStatus(). Сервисы координируются с репозиториями и моделями; логика транзакций (через DB::transaction()) инкапсулируется в методы сервиса.

4. Валидация входящих данных через FormRequest.
   Классы запросов форм в Laravel (App\Http\Requests\*) для проверки ввода и авторизации.

Отделяет логику проверки от контроллеров, что снижает сложность кода.

Обеспечивает централизованное хранение правил проверки и сообщений об ошибках.

Поддерживает автоматический HTTP-ответ при сбое проверки.

Объединяет проверки авторизации (метод authorize()) с валидацией.

Реализация: пользовательские классы запросов (например, StoreRequestRequest, AssignMasterRequest) определяют правила и логику авторизации; в методах контроллера указывается тип запросов, что запускает автоматическую проверку; сообщения об ошибках возвращаются в формате JSON или перенаправляются с данными flash‑сессии.

5. Enums (App\Enums\*) для  представления статусов,  кодов ошибок и ролей пользователей.

Обеспечивает безопасность ввода и предотвращает появление недопустимых значений статусов и ролей.

Централизует определения и описания статуса в одном месте.

Улучшает читаемость кода и удобство сопровождения.

Перечисление ApiError стандартизирует коды ошибок и удобные для пользователя сообщения; 

6. Управление доступом на основе ролей (RBAC), используя встроенные в Laravel функции аутентификации и авторизации.

Применяет политики безопасности на уровне приложения.

Позволяет детально управлять разрешениями с помощью ролей (например, мастер, диспетчер).

Поддерживает проверку доступа на уровне маршрута и действий.

Интегрируется с промежуточным программным обеспечением для надежной защиты маршрутов.

Реализация: роли пользователей хранятся в базе данных и управляются с помощью средств настройки; промежуточное программное обеспечение (например, EnsureUserHasRole::class . ':dispatcher|master') защищает маршруты; 

7. Транзакции базы данных для критически важных операций: использовать обновление статуса с блокировкой (lockForUpdate()) транзакционно.

Обоснование:
Предотвращает возникновение конфликтных ситуаций, когда несколько пользователей изменяют один и тот же Request.

Обеспечивает согласованность данных при сложных переходах состояний.

Использует блокировку на уровне базы данных для атомарных операций.

Соответствует реальным рабочим процессам (например, только один пользователь может отправить результативный запрос).

Реализация: UpdateStatus($data, $id, $status, $lock = false) принимает флаг $lock; когда значение $lock равно true, DB::transaction() завершает запрос с помощью lockForUpdate();
